# RPA-робот для поиска книг (Book Price Aggregator)

## 1. Описание проекта

Данный проект представляет собой RPA-робота, разработанного для автоматизации сбора информации о книгах из интернет-магазинов. Робот принимает на вход список названий книг, выполняет поиск по заданным торговым площадкам, извлекает ключевую информацию и формирует итоговый отчет.

**Цель разработки:** Демонстрация навыков проектирования и разработки RPA-решений с соблюдением лучших практик (Best Practices) и принципов программирования.

## 2. Функциональные требования (выполнено)

Согласно техническому заданию, робот реализует следующий функционал:

1.  **Чтение входных данных:** Принимает список книг из Excel-файла (столбец "Название книги" обязателен).
2.  **Поиск по магазинам:** Выполняет поиск по одному или нескольким книжным магазинам (пример: `https://www.chitai-gorod.ru/`).
3.  **Сбор данных:** Для каждой найденной книги извлекает:
    *   Прямую ссылку на книгу.
    *   Тип обложки (твердая/мягкая).
    *   Текущую цену.
4.  **Формирование отчета:** Сохраняет итоговую таблицу с собранными данными в папку `Output` в формате Excel.
5.  **Обработка ошибок:** Реализована логика повторных попыток (retries) при временных сбоях и механизм перенаправления проблемных транзакций в отдельную папку для дальнейшего разбора (Fail).

## 3. Архитектура решения

Проект построен на базе шаблона **Reframework**. Получение входных данных и их обработка осуществляется в одном проекте. С наличием лицензии на оркестратор можно будет использовать паттерн "Диспетчер-Исполнитель" для обеспечения масштабируемости

### 3.1. Технические детали и обоснование решений

*   **Платформа:** Primo RPA
*   **Обработка транзакций:**
    *   Каждая строка входного файла (название книги) рассматривается как отдельная транзакция.
    *   Реализован механизм повторных попыток (Retry). В случае неудачного поиска (например, таймаут сайта) робот выполняет заданное количество попыток перед тем, как признать транзакцию неуспешной.
    *   Для транзакций, которые не удалось обработать после всех попыток, предусмотрено логирование и выгрузка в отдельную папку (Fail). Это позволяет бизнес-пользователю вручную разобрать проблемные позиции.
*   **Работа с почтой:**
    *   **Нюанс:** В текущей реализации отправка отчета по электронной почте заменена на локальное сохранение файла.
    *   **Причина:** Тестовые почтовые серверы (Gmail, Yandex) наложили ограничения на тестовые учетные записи, что не позволило стабильно реализовать SMTP-клиент.
    *   **Рекомендация для продакшена:** Для отправки отчета необходимо использовать корпоративный почтовый сервер с корректно настроенными SMTP-параметрами, которые должны храниться в защищенном хранилище (Assets) Оркестратора.

## 4. Требования к окружению и запуску

### 4.1. Входные данные
*   Формат файла: Microsoft Excel (.xlsx).
*   Структура файла: Таблица с обязательным заголовком **"Название книги"**.
    *Пример:*

    | Название книги               |
    |-------------------------------|
    | Война и мир                   |
    | Преступление и наказание      |
    | Мастер и Маргарита            |

### 4.2. Настройка конфигурации
Все ключевые настройки вынесены в файл `Config.xlsx` для упрощения администрирования и соответствия Best Practices:
*   Пути к входным и выходным файлам.
*   URL-адреса магазинов.

**Рекомендация:** В случае развертывания на Оркестраторе, критичные параметры (логины/пароли от почты, строки подключения) должны быть перенесены в **Assets**.

## 5. Логика работы робота (Flow)

1.  **Инициализация:** Робот запускается, считывает конфигурацию из `Config.xlsx`, открывает необходимые приложения (Excel, браузер).
2.  **Диспетчер (GetTransactionData):** Читает входной Excel-файл из папки `Input`. Преобразует каждую строку в транзакцию.
3.  **Обработка (Process):**
    *   Для каждой транзакции (книги) вызывается компонент `Process.ltw`.
    *   Робот переходит на сайт магазина, выполняет поиск по названию книги.
    *   Извлекает со страницы результатов необходимые данные: ссылка, цена, обложка.
    *   Данные складываются в выходную таблицу.
4.  **Завершение транзакции:**
    *   **Успех:** Переход к следующей книге.
    *   **Ошибка (временная):** Повторная попытка (Retry).
    *   **Ошибка (критическая):** Логирование ошибки, запись данных о книге и причине ошибки в отдельный файл в папку `Fail`. Транзакция пропускается.
5.  **Завершение работы:** После обработки всех книг вызывается компонент `UpdateStatusFile.ltw`, который формирует итоговый Excel-отчет в папку `Результат_ddMMyyyy HHmmss.xlsx`. Робот закрывает все приложения и завершает работу.

## 6. Планы по улучшению (Roadmap)

*   **Интеграция с Оркестратором:** Подключение к Оркестратору для централизованного управления, мониторинга и логирования.
*   **Использование очередей:** Замена паттерна Диспетчер-Исполнитель на нативные очереди Оркестратора. Это позволит получать статус по каждой транзакции в реальном времени и упростит мониторинг.
*   **SMTP-отправка:** Настройка отправки итогового отчета на почту с использованием корпоративных учетных данных, хранящихся в Assets Оркестратора.
*   **Поддержка нескольких магазинов:** Добавление компонентов для парсинга других площадок (например, Лабиринт) и объединение результатов.

## 7. Демонстрация решения

На собеседовании будет представлена:
1.  **Презентация:** Обзор архитектуры и обоснование принятых проектных решений.
2.  **Демонстрация кода:** Разбор ключевых компонентов, демонстрация читаемости, структурированности и обработки ошибок.
3.  **Запуск проекта:** Демонстрация работы робота "здесь и сейчас" с предоставлением входного файла и выводом итогового отчета в папку `Output`.
